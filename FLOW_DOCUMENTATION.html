The core feature. User pastes chapter text or uploads textbook images; the system returns a structured lesson with explanation, examples, visual briefs, a Mermaid diagram, and a follow-up topic list.

Request (Text mode)

POST /api/v1/explain/text-form Content-Type: multipart/form-data

output_language: "english"  # english | hindi | roman_hindi
topic_hint:      "Newton's Laws"  # optional

Request (Image mode)

POST /api/v1/explain/images Content-Type: multipart/form-data 
files[]:         [page1.png, page2.png]  # PNG or JPEG
output_language: "hindi"
topic_hint:      ""
stack:           "openai"

Response

200 ExplainResponse 
{
  "explanation_markdown": "## Newton's First Law\n\n...",
  "simple_examples": ["A ball rolling on a frictionless surface..."],
  "visual_briefs": [
    {
      "title":            "Force Diagram",
      "description":      "Arrow vectors showing F = ma",
      "kind":             "diagram",
      "suggested_format": "infographic"
    }
  ],
  "suggested_followup_topics": ["Friction", "Momentum"],
  "video_lesson_prompt":       "An animated physics classroom...",
  "mermaid_diagram":           "graph TD\n  F[Force] --> A[Acceleration]",
  "output_language_used":      "english"
}

Internal Flow (Step by Step)

FE
Frontend — api.ts

User clicks Generate. explainFromText() or explainFromImages() is called.

Builds a FormData object with chapter text (or image File objects), language, topic hint, and stack. Sends via fetch().

BE
Backend — endpoints/explain.py

FastAPI endpoint receives the form data. Normalizes output_language string, maps stack → llm_provider ("openai"→"openai", "opensource"→"ollama").

For image endpoints, reads each uploaded file into bytes and passes them to the service.

SV
Service — services/explanation.py

Builds a system prompt instructing the LLM to act as a teacher with output language constraints.

Builds a user message containing the chapter text or base64-encoded images + instructions to return a strict JSON schema.

Calls llm_provider.complete_chat(messages, max_tokens=4096, temperature=0.35).

AI
LLM Provider — llm/openai_provider.py or ollama_provider.py

OpenAI: Calls chat.completions.create(model="gpt-4o-mini", response_format={"type":"json_object"}). Returns raw JSON string.

Ollama: POSTs to http://localhost:11434/api/chat with format="json". Uses llava model for image inputs.

SV
Service — Post-processing

Parses the raw JSON string into an ExplainResponse Pydantic model.

Sanitizes mermaid_diagram: strips code fences, normalizes unicode, validates Mermaid header keyword.

RE
Response → Frontend

FastAPI serializes ExplainResponse to JSON and returns HTTP 200.

Frontend: renders explanation_markdown with ReactMarkdown, passes mermaid_diagram to MermaidDiagram component, lists visual_briefs as cards, and auto-triggers the downstream pipeline (Quiz → Slides → Flashcards → Infographic → Audio).

# Returns binary PPTX file

Internal Flow

SV
services/slides.py — LLM phase

Prompts LLM: "You are a teacher preparing slides. Return JSON with deck_title and a slides array." Calls LLM with max_tokens=8000.

Parses LLM JSON → list of SlideItem (title, bullets, speaker_notes). Normalizes bullet count (3–6 bullets per slide).

SV
services/slides.py — PPTX build phase

Uses python-pptx: creates Presentation(), adds title slide, then one bullet slide per SlideItem. Adds speaker notes to each slide.

Exports to bytes via io.BytesIO. Generates deck_id = "sdeck_" + UUID[:16]. Stores _decks[deck_id] = (pptx_bytes, filename) in module-level dict.

RE
Download Flow

Frontend calls fetchSlideDeckBlob(deck_id). Backend looks up _decks[deck_id] and returns a StreamingResponse with the PPTX bytes.

Browser receives the binary file and triggers OS download dialog.

Internal Flow

SV
services/tts.py — Text cleaning

Strips markdown (headers, bold, code blocks), collapses whitespace → produces clean plain-text narration script.

AI
Translation (conditional)

If output_language != "english": calls LLM to translate the cleaned text into the target language (Hindi Devanagari or Roman Hindi).

Validates translation by checking script ratio (Devanagari character percentage). Sets translation_applied=True.

SV
Truncation

Truncates to safe limits: OpenAI TTS → 4,000 chars; edge-tts → 50,000 chars. Sets truncated=True if text was cut.

EX
TTS synthesis

OpenAI: client.audio.speech.create(model, voice, input_text) → MP3 bytes.

edge-tts: Microsoft cloud TTS, free, supports 100+ languages. Calls synthesize_speech(voice, text) → MP3 bytes.

RE
Download + Playback

MP3 bytes cached at _audio_store[audio_id]. Frontend calls GET /audio/{audio_id}/file → StreamingResponse with MP3 bytes.

Browser creates a Blob URL, sets it as <audio src=...> and calls audio.play().

Internal Flow by Stack

BE
endpoints/video.py — routing

Checks stack: routes to video_openai or video_opensource module.

Also checks chain_target_seconds: if ≥ 18s with OpenAI, uses chained video creation.

EX
OpenAI Sora path

Calls client.videos.create_and_poll(prompt, model, seconds, size, input_reference) — Sora API.

Chaining: Creates 12s video → calls videos.extend() with 4/8/12s segments until reaching chain_target_seconds. Final MP4 downloaded from OpenAI S3.

SV
Local animation path (video_opensource.py)

Creates LocalVideoJob, starts async background task.

Uses PIL to generate frames: detects topic keywords in prompt (mirrors, optics, biology), draws domain-specific sketches (optical diagrams, cell sketches, etc.).

Applies animation effects: rotation, translation, gradient panning (Ken Burns effect for motion_plus).

Encodes frames to MP4 using imageio. Stores bytes in job cache.

FE
Frontend — Polling Loop

Calls GET /video/jobs/{id} every 2–3 seconds. Checks status.

Shows live progress bar (local jobs). When status == "completed", enables download button.

RE
Download + Playback

Frontend calls GET /video/jobs/{id}/file.

Local: returns cached MP4 bytes from _jobs[id].mp4_bytes.
OpenAI: downloads MP4 from Sora's S3 URL and streams to browser.

Browser plays in <video> element or saves as .mp4.

⛓ Auto-Pipeline — Client-Side Orchestration

After the Explain response is received, the frontend automatically triggers all other features in sequence. This entire pipeline is orchestrated client-side in App.tsx using async/await.

// Simplified from App.tsx
const result = await explainFromText(text, lang, hint, stack);
setState({ explanation: result });

// Auto-pipeline (sequential, each awaits the previous)
await generateQuiz({
  topic: derivedTopic,
  context_text: result.explanation_markdown + result.simple_examples.join(''),
  output_language: lang,
  stack,
  question_count: 5,
  difficulty: "standard"
});

await generateSlideDeck({ topic, context_text, output_language: lang, stack, slide_count: 8 });
await generateFlashcards({ topic, context_text, output_language: lang, stack, card_count: 12 });
await generateInfographic({ topic, context_text, output_language: lang, stack });
await generateSpeech({ context_text, output_language: lang, stack });
Note: Each step in the pipeline uses the explanation content (markdown + examples) as context_text, so all generated assets (quiz, slides, flashcards) are grounded in the same lesson material.

🤖 LLM Provider Abstraction

All LLM calls go through a Protocol-based interface. A factory function selects the right provider based on the request's stack param.

# app/services/llm/protocol.py
class LLMProvider(Protocol):
    async def complete_chat(
        self,
        messages: list[Message],
        *,
        max_tokens: int = 4096,
        temperature: float = 0.4
    ) -> str: ...  # Returns raw JSON string

# app/services/llm/factory.py
def get_llm_provider(vision: bool = False, provider: str = None) -> LLMProvider:
    selected = provider or settings.llm_provider
    if selected == "openai":
        return OpenAIProvider(settings)
    return OllamaProvider(settings, vision=vision)

OpenAI Provider

  • Model: gpt-4o-mini (configurable)
  • JSON mode: response_format="json_object"
  • Vision: image URLs in messages
  • Requires: OPENAI_API_KEY

Ollama Provider

  • Default model: llama3.2
  • Vision model: llava
  • JSON mode: format="json"
  • Local: http://localhost:11434

Image Providers

  • OpenAI: DALL-E 3
  • Opensource: SD WebUI (AUTOMATIC1111)
  • Both return base64 PNG
  • SD WebUI: localhost:7860
🌐 Language Support

Every LLM call is parameterized by output language. The same topic can be fully generated in three language modes.

ValueScriptTTS behaviorExample
englishLatinNo translation needed"An object at rest stays at rest"
hindiDevanagariLLM translates to Devanagari, then TTS synthesizes"एक स्थिर वस्तु स्थिर रहती है"
roman_hindiLatin (Hinglish)LLM translates to Roman Hindi"Ek sthir vastu sthir rahti hai"
Implementation: All service prompt builders include strict language instructions like "All text output MUST be in Hindi (Devanagari script). Do NOT use English except for proper nouns." The TTS service independently validates the translation by measuring the Devanagari character ratio before synthesizing audio.
💾 File Caching — In-Memory Store

There is no database. Generated binary files (PPTX, MP3, MP4) are cached in module-level Python dicts and served by ID. Caches are cleared on server restart.

Cache variableLocationKeyValue
_decksservices/slides.pydeck_id ("sdeck_…")(pptx_bytes, filename)
_audio_storeservices/tts.pyaudio_id ("aid_…")(mp3_bytes, filename, mime_type)
_jobsservices/video_opensource.pyjob_id ("osvid_…")LocalVideoJob object with MP4 bytes
Production note: Replace in-memory caches with object storage (AWS S3, GCS) and a database (PostgreSQL) for multi-instance deployments. The current architecture is designed for single-instance / demo use.
